# Work Daily Report

把一组工作日常记录整理成中文 Markdown 周报、月报或年报。**本 SKILL 是结构与规则的唯一权威**——调用方只负责喂输入变量，不要再额外塞强制结构。

## Input

调用方提供：

- `title`：报表标题（如 `牛马日常月报 - 2026-05`）。
- `type`：`week` / `month` / `year`。
- `report_type_label`：`周报` / `月报` / `年报`。
- `style`：本次报表的风格提示（详见下方 Type Style）。
- `platform_count` / `record_count`：本阶段平台数、记录数。
- `platforms`：本阶段涉及的平台名列表（已合并）。
- `source`：按平台分组的原始记录（每条 `- {date}: {content}`）。

## Output

**直接输出 Markdown，不要用 ``` 或 ```markdown 把整篇报表包起来**。结构：

```markdown
# {title}

## 概览

{用 80-180 字说明这是一份 {report_type_label}，统计本阶段主要在哪些平台或方向投入，形成了哪些产出。}

## 平台总结

### {平台名称}

- {3-6 条合并后的工作产出，每条 1-2 行短句，不逐条复述原始记录}

## 学习收获

- {2-5 条，从记录中提炼学到的方法、框架、业务理解或工程经验}

## 需要加强

- {2-5 条，从记录中看出的知识短板、能力短板或后续要补齐的方向}

## 下一阶段计划

- {2-5 条，基于当前记录自然推出的下一步计划；如果原始记录不足，只写稳妥的延续动作，不编造具体项目}

## 结束语

{1-2 句，第一人称视角自然收尾，语气务实。}
```

## Rules

### 结构与排版
- 恰好一个 H1；H1 与 `title` 完全一致。
- 不要用 ``` 包裹整个输出；正文里原有的代码块、文件路径、PR 编号、commit hash 等原文标识**保持原样**，不要改写或删掉。
- 平台不要重复出现 H3；同名平台合并。
- 「平台总结」**按本阶段产出密度从多到少排序**（字数或记录条数权衡），主力平台在前。
- 不要逐条复述原始记录；不要逐日罗列日期。
  - 周报：允许在「平台总结」或「学习收获」里点出 1-2 个关键节点的具体日期（发布日、踩坑日等）。
  - 月报 / 年报：不出现具体日期，只在概览里说本月 / 本年。

### 内容真实性
- 学习收获、需要加强、下一阶段计划**只能从提供的记录中推导**，不编造记录外的具体项目、人名、数据、收益数字。
- PR 编号（如 `blog#20`）、commit hash、issue 链接是后续追溯的关键索引，至少在「平台总结」对应条目里保留一次。

### 记录较少时的精简策略
- 当 `record_count < 3`：可以精简到「概览 + 平台总结 + 结束语」三段，不要硬凑「学习收获 / 需要加强 / 下一阶段计划」。
- 当某一项实在推不出来（如本周纯运维、无新学习点）：该段写一句兜底，如「本阶段以稳定运维 / 迭代为主，未触发新的能力短板」，不要编造。

### 结束语防泄漏（高频翻车区，专门强调）
- 结束语**严禁出现**：「AI」「模型」「本次总结」「报表」「以上是…」「根据您提供的」「希望对您有帮助」「如有需要…」「让我们…」等元描述或客套语。
- 结束语用**第一人称**（我 / 咱们），不要写「用户」「您」。
- 不要在正文任何地方提到 skill、prompt、模型、生成流程；如果原始记录里写到「使用 skill」「整理 prompt」等，请用「总结结构」「报表规则」之类业务化措辞替代。

### 空记录兜底
- 如果没有日志，**只输出**：

```markdown
# {title}

暂无记录。
```

## Type Style

- `week`：强调本周重点进展、已解决问题、阻塞点和下周可延续动作。
- `month`：强调完成模块、修复问题、体验改进、可见产出和月度学习模式。
- `year`：强调年度成果、关键项目、长期改进、经验沉淀和下一年方向。
