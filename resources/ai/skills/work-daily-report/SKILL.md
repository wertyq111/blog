# Work Daily Report

把一组工作日常记录整理成中文 Markdown 周报、月报或年报。**本 SKILL 是结构与规则的唯一权威**——调用方只负责喂输入变量，不要再额外塞强制结构。

## Input

调用方提供：

- `title`：报表标题（如 `牛马日常月报 - 2026-05`）。
- `type`：`week` / `month` / `year`。
- `report_type_label`：`周报` / `月报` / `年报`。
- `style`：本次报表的风格提示（详见下方 Type Style）。
- `platform_count` / `record_count`：本阶段平台数、记录数。
- `platforms`：本阶段涉及的平台名列表（已合并）。
- `source`：按平台分组的原始记录（每条 `- {date}: {content}`）。
- `previous_summary`（可选）：上一期同类型报表的「概览」段。**未提供时整篇不得出现任何"相比上期 / 环比"措辞。**

## Output

**直接输出 Markdown，不要用 ``` 或 ```markdown 把整篇报表包起来**。结构按以下顺序：

```markdown
# {title}

## 🏝️ 概览

{用 80-180 字说明这是一份 {report_type_label}，统计本阶段主要在哪些平台或方向投入，形成了哪些产出。若提供 previous_summary，在末尾用 1-2 句点出与上期的延续或变化。}

## 🥬 数据速览

| 指标 | 数值 |
| --- | --- |
| 覆盖平台 | {platform_count} 个 |
| 记录条数 | {record_count} 条 |
| 活跃天数 | {source 里去重后的日期数} 天 |
| 关键交付 | {记录中出现的 PR / commit / issue 引用条数} 项 |

## ✨ 本阶段亮点

> 1. **{关键词}**：{一句话，挑产出密度或影响面最大的事}
> 2. **{关键词}**：{一句话}

## ✈️ 平台总结

### {平台名称}

- **{动词开头}**{合并后的工作产出，每条 1-2 行短句，3-6 条，不逐条复述原始记录}

## 🌱 学习收获

- {2-5 条，从记录中提炼学到的方法、框架、业务理解或工程经验}

## 🦴 需要加强

- {2-5 条，从记录中看出的知识短板、能力短板或后续要补齐的方向}

## 🗺️ 下一阶段计划

- [ ] {2-5 条，checklist 格式，基于当前记录自然推出的下一步计划；如果原始记录不足，只写稳妥的延续动作，不编造具体项目}

## 📜 关键项目时间线

- {月份/季度}：{里程碑}（**仅 year 输出本节**，3-8 条）

## 🌙 结束语

{1-2 句，第一人称视角自然收尾，语气务实。}
```

## Rules

### 结构与排版
- 恰好一个 H1；H1 与 `title` 完全一致，**H1 不加图标**。
- H2 节标题的图标固定如上（🏝️ 概览 / 🥬 数据速览 / ✨ 本阶段亮点 / ✈️ 平台总结 / 🌱 学习收获 / 🦴 需要加强 / 🗺️ 下一阶段计划 / 📜 关键项目时间线 / 🌙 结束语），不增减、不替换。
- 平台 H3 **不加图标**；同名平台合并，不重复出现 H3。
- 「平台总结」**按本阶段产出密度从多到少排序**（字数或记录条数权衡），主力平台在前；条目尽量以**加粗动词**开头（如 **重构**、**修复**、**打通**）。
- 「本阶段亮点」2-3 条，blockquote 编号列表，关键词加粗；与平台总结允许内容重叠，但不要原句重复。
- 「数据速览」**只能从输入统计**：活跃天数 = `source` 去重日期数；关键交付 = 记录里 PR / commit / issue 引用条数。某指标统计不出来就**整行省略**，不写 0、不猜。
- 「下一阶段计划」用 `- [ ]` checklist 格式，方便直接复制进待办列表。
- 「关键项目时间线」**仅 `type` 为 `year` 时输出**；周报 / 月报省略整节。
- 不要用 ``` 包裹整个输出；正文里原有的代码块、文件路径、PR 编号、commit hash 等原文标识**保持原样**，不要改写或删掉。
- 不要逐条复述原始记录；不要逐日罗列日期。
  - 周报：允许在「平台总结」或「学习收获」里点出 1-2 个关键节点的具体日期（发布日、踩坑日等）。
  - 月报 / 年报：不出现具体日期，只在概览里说本月 / 本年（年报时间线里的月份 / 季度除外）。

### 内容真实性
- 学习收获、需要加强、下一阶段计划**只能从提供的记录中推导**，不编造记录外的具体项目、人名、数据、收益数字。
- PR 编号（如 `blog#20`）、commit hash、issue 链接是后续追溯的关键索引，至少在「平台总结」对应条目里保留一次。
- 环比叙述**仅当提供 `previous_summary` 时**出现，且只比较两期概览都明确提到的事实；禁止编造增减幅度或数字。

### 记录较少时的精简策略
- 当 `record_count < 3`：精简为「🏝️ 概览 + ✈️ 平台总结 + 🌙 结束语」三段，其余节（含数据速览、亮点、计划、时间线）全部省略，不要硬凑。
- 当某一项实在推不出来（如本周纯运维、无新学习点）：该段写一句兜底，如「本阶段以稳定运维 / 迭代为主，未触发新的能力短板」，不要编造；「本阶段亮点」推不出 2 条就省略整节。

### 结束语防泄漏（高频翻车区，专门强调）
- 结束语**严禁出现**：「AI」「模型」「本次总结」「报表」「以上是…」「根据您提供的」「希望对您有帮助」「如有需要…」「让我们…」等元描述或客套语。
- 结束语用**第一人称**（我），不要写「用户」「您」。
- 不要在正文任何地方提到 skill、prompt、模型、生成流程；如果原始记录里写到「使用 skill」「整理 prompt」等，请用「总结结构」「报表规则」之类业务化措辞替代。

### 空记录兜底
- 如果没有日志，**只输出**：

```markdown
# {title}

暂无记录。
```

## Type Style

- `week`：强调本周重点进展、已解决问题、阻塞点和下周可延续动作。
- `month`：强调完成模块、修复问题、体验改进、可见产出和月度学习模式。
- `year`：强调年度成果、关键项目、长期改进、经验沉淀和下一年方向。
